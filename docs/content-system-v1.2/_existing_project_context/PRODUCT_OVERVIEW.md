# Context Reader — обзор продукта и функционала

> Этот документ написан для передачи внешнему инструменту/модели (например, другому
> GPT) в качестве контекста для брейнштормa — новых фич, улучшений, направлений
> развития. Он не заменяет `PLAN.md`/`DESIGN.md`/`AI_PIPELINE.md`/`PROGRESS.md` —
> собирает суть из них в одном месте, подробно и без внутреннего жаргона проекта.
> Актуален на 2026-07-22.

## 1. Что это за продукт

**Context Reader** — интерактивный ридер для изучения иностранного языка через
чтение связного текста (не карточки, не отдельные упражнения). Пользователь читает
статью/рассказ на изучаемом языке, слушает синхронную озвучку, и в любой момент
может ткнуть в незнакомое слово или фразу — снизу экрана выезжает панель
(«Bottom Sheet») с объяснением: что это значит именно в этом предложении, как
произносится, какая грамматика за этим стоит.

Идея в противовес классическим языковым приложениям (Duolingo-подобным): никакой
геймификации, стриков, XP, обучающей цветовой разметки поверх текста. Текст выглядит
как обычная статья, пока пользователь сам не попросит объяснение конкретного места.
Референсы, на которые ориентировались при дизайне: Readwise Reader, Matter,
Instapaper (минимализм чтения) — и осознанное отталкивание от LingQ/EWA (перегруженность
цветовой/учебной разметкой).

Технически: React 19 + TypeScript + Vite на фронтенде, Vercel serverless functions
как тонкий backend только для AI-генерации (TTS, текст, аннотации, перевод), без
отдельной постоянной базы данных — сгенерированные уроки складываются как JSON-файлы
в Vercel Blob storage.

## 2. Для кого

Целевой пользователь — сам создатель проекта, изучающий иностранный язык (сейчас
реализовано и проверено на слух для французского; немецкий/английский/греческий
поддержаны технически, но озвучка для них не проверена на слух). Уровень текста —
A1–C1 настраиваемый (дефолт A2), учебная метафора — «читаю настоящий текст, а не
упражнение», подсказки — на русском (родной язык пользователя).

## 3. Из чего состоит опыт пользователя (end-to-end сценарий)

1. **Библиотека** (`LibraryPage`) — стартовый экран. Список уроков: один встроенный
   пример-урок (полностью вручную написанный и размеченный French A2–B1 текст) плюс
   все ранее сгенерированные уроки (подтягиваются с бэкенда). Карточка урока показывает
   язык, уровень, оценку времени чтения, каким провайдером озвучено. Кнопка
   «+ Новый урок» ведёт к генерации.

2. **Генерация нового урока** (`GenerateLessonPage`) — форма:
   - Материал: либо тема (AI сам пишет текст на эту тему), либо вставленный пользователем
     готовый текст (AI адаптирует его под уровень).
   - Язык: fr / de / en / el.
   - Уровень (A1–C1) и объём (100–250 слов).
   - Провайдер озвучки — временный A/B-переключатель ElevenLabs vs OpenAI+Whisper
     (не постоянный UX-элемент, останется только один после того, как выбор
     подтвердится статистически, а не только на слух).
   - Прогресс генерации показывается по стадиям (текст → токенизация → аннотации →
     озвучка+тайминги → сохранение) — процесс занимает пару минут, не секунды.
   - Результат — сразу открывается в ридере.

3. **Ридер** (`ReaderPage`) — основной экран:
   - Заголовок урока, метаданные (язык · уровень · длительность), кнопка назад,
     кнопка настроек.
   - Текст статьи — крупная типография, много воздуха, без всякой преждевременной
     разметки слов до первого взаимодействия.
   - Плеер озвучки, всплывающий снизу как компактная sticky-панель после старта
     чтения: play/pause/stop, регулятор скорости (5 значений, 0.6×–1.5×), индикатор
     прогресса, «продолжить с этого места».
   - Синхронная karaoke-подсветка произносимого слова во время воспроизведения
     (мягкий «маркерный» фон, не резкий UI-акцент).
   - Клик по любому слову → Bottom Sheet снизу с объяснением (см. раздел 4) — можно
     кликать и во время паузы, и во время воспроизведения.
   - Режим перевода предложений — отдельный тумблер в настройках: показывает построчный
     перевод всего предложения (не то же самое, что объяснение отдельного слова в
     Bottom Sheet). Для встроенного урока переводы уже готовы, для сгенерированных —
     подгружаются лениво по запросу.
   - Настройки: тема (light/dark), размер шрифта, режим перевода.
   - «Сохранить» — пока стаб на localStorage (не полноценная библиотека избранного).

4. **Bottom Sheet — объяснение слова/фразы** (ядро продукта, сейчас в процессе
   существенного редизайна — см. раздел 7):
   - Клик по слову резолвится в конкретный токен текста (не «примерно то же слово
     где-то ещё» — каждое вхождение слова в тексте — это свой токен с собственным
     объяснением, учитывающим контекст именно этого предложения).
   - Короткий ответ появляется сразу: часть речи, форма как она стоит в тексте,
     перевод в контексте, кнопка озвучить именно эту форму, само предложение с
     переводом (с подсветкой того, что выбрано).
   - «Подробнее» — по явному запросу подгружается более глубокий разбор: как это
     работает грамматически, сравнение форм/времён, таблица спряжений/склонений,
     похожие конструкции, грамматическая справка — но не жёсткий фиксированный
     набор карточек, а только те секции, которые реально добавляют знание для
     конкретного слова (у служебного слова не будет спряжения, у существительного —
     сравнения времён, и т.д.).
   - Составные конструкции (фразовый глагол, предлог+артикль, возвратный глагол
     с частицей) объясняются как единое целое, но клик остаётся по конкретному
     слову — не подменяет цель клика на всю фразу.

## 4. Данные и модель контента

Каждый урок (`Lesson`) — это:
- метаданные (язык, уровень, заголовок/перевод заголовка, оценка времени чтения);
- текст, разбитый на параграфы → предложения → токены (слово или пунктуация),
  с id для каждого уровня;
- у каждого токена — опциональные `startTime`/`endTime` (для синхронной подсветки
  под озвучку);
- аннотации — по одной на каждый значимый токен (`Annotation.id === token.id`),
  двухуровневая структура: `summary` (всегда есть, для короткого состояния Bottom
  Sheet) + `details` (подгружается лениво при «Подробнее»);
- какой TTS-провайдер использован и отчёт о качестве выравнивания озвучки
  (`alignmentReport`) — сохраняется вместе с уроком, чтобы можно было
  постфактум понять, насколько надёжна подсветка, не переслушивая всё заново.

Токенизация и разбиение на предложения — не наивный regex, а `Intl.Segmenter` с
учётом языка (важно для сокращений, кавычек, границ предложений на разных языках).

## 5. AI-пайплайн генерации урока (что происходит «под капотом» за эти пару минут)

Шаги (каждый — отдельный AI-вызов или детерминированный код):

1. **Генерация/адаптация текста** — по теме или на основе вставленного пользователем
   текста, под нужный уровень и объём.
2. **Токенизация** — детерминированная (не AI), разбивает готовый текст на
   параграфы/предложения/токены через `Intl.Segmenter`.
3. **Генерация аннотаций** — для каждого значимого токена отдельный AI-вызов
   (сейчас пакетно, не по клику пользователя — дешевле и позволяет проверить
   качество один раз до показа пользователю), с полным предложением как контекстом
   (не парой соседних слов — иначе не хватает смысла объяснить, например, выбор
   imparfait/passé composé). Структурированный JSON-ответ по строгой схеме, не
   свободный текст.
4. **Озвучка + word-level тайминги** — два поддерживаемых провайдера:
   - **ElevenLabs** (сейчас дефолт) — один вызов `with-timestamps`: тайминги — не
     пост-анализ, а побочный продукт самого синтеза речи. Голос — Matilda,
     `eleven_multilingual_v2`, выбран на слух из нескольких кандидатов.
   - **OpenAI TTS + Whisper** (fallback) — TTS не даёт своих таймингов, поэтому
     отдельный ASR-проход (Whisper) слушает готовое аудио и восстанавливает
     сопоставление слов эвристикой.
   - Поверх обоих — универсальный **recovery-слой**: если какие-то слова не
     получили валидный тайминг напрямую от провайдера, они интерполируются между
     соседними размеченными словами, микро-паузы растягиваются, нарушения
     порядка (не по возрастанию времени) клампятся.
   - **Quality gate**: после восстановления считается отчёт — сколько слов
     получили тайминг напрямую vs через восстановление. Если прямое покрытие
     ниже 95% или доля восстановленных выше 10% — урок не сохраняется, пользователь
     видит причину текстом (раньше это тихо проглатывалось — баг, найденный
     внешним аудитом и исправленный).
5. **Сохранение** — итоговый `Lesson` (текст + аннотации + тайминги + отчёт о
   качестве) пишется в Vercel Blob, индекс библиотеки обновляется.

Отдельно — **озвучка одного слова по клику** в Bottom Sheet: не нарезка общей
дорожки урока (это давало артефакты коартикуляции — слово «découvrir» звучало
обрубленным как «couvrir»), а отдельная короткая генерация тем же провайдером и
голосом, что и весь урок, с кэшированием по хэшу параметров — повторный клик не
делает повторный TTS-вызов.

## 6. Технический стек и архитектурные принципы

- **Фронтенд**: React 19 + TypeScript + Vite, без серверного рендеринга.
- **Backend**: Vercel serverless functions (`api/*.ts`) — только для операций,
  требующих серверных API-ключей (TTS, генерация текста/аннотаций, перевод,
  сохранение/чтение уроков). Ключи никогда не уходят в браузер.
- **Хранилище**: Vercel Blob — уроки как JSON-файлы + сгенерированные mp3, не
  реляционная БД. Простая read-модель (индекс библиотеки), без миграций схемы.
- **Провайдеро-независимость через интерфейсы**: `NarrationAdapter` абстрагирует
  озвучку — `BrowserSpeechAdapter` (Web Speech API, ранний прототип, ещё в кодовой
  базе как запасной вариант) и `PrecomputedAudioAdapter` (проигрывание готового
  файла с таймингами) реализуют один и тот же контракт. Аналогично TTS-провайдеры
  (OpenAI/ElevenLabs) и языки (`LanguageConfig`) закладываются за конфигурацией,
  не хардкодятся в компоненты — сделано осознанно, чтобы можно было добавить
  провайдера/язык, не переписывая логику ридера.
- **Тестирование**: vitest (unit-тесты на детерминированные части пайплайна —
  токенизация, сопоставление таймингов символов с токенами, recovery-слой,
  quality gate), oxlint, tsc для типов. UI проверяется вживую в браузере
  (`npm run dev`), не через e2e — библиотека Web Speech API и реальная озвучка
  всё равно требуют ручной проверки на слух.
- **Мультиязычность**: контракт (`LanguageCode = 'fr' | 'de' | 'en' | 'el'`) готов
  на уровне типов, токенизации и API — но озвучка для de/en/el использует тот же
  голос ElevenLabs, что и французский (модель мультиязычная и технически говорит на
  всех четырёх), и качество на слух пока не проверено человеком — явно помечено
  предупреждением в UI генерации.

## 7. Что сейчас в работе (на момент написания документа)

Идёт переработка Bottom Sheet по внешнему дизайн-концепту (`greek-bottom-sheet-
handoff/` — согласованный на гречском примере продуктовый контракт: короткое
состояние без лишних полей вроде «словарная форма», секции подробностей не
фиксированный набор карточек, а только релевантные части речи, сравнение времён
глагола на одном лице, визуальные правила без CAPS/тяжёлого bold/серых карточек
вокруг таблиц). Также только что реализована и провалидирована архитектура
озвучки v2 (см. раздел 5) — recovery-слой, quality gate, клипы отдельных слов,
мультиязычный контракт.

## 8. Что сознательно НЕ входит в продукт (антипаттерны, от которых отталкивались)

- Никакой постоянной цветовой разметки слов по статусу изучения (как в LingQ) —
  текст выглядит как обычная статья.
- Никакой геймификации — стриков, очков, значков.
- Bottom Sheet — не полноэкранная модалка, а частичная панель снизу (стандарт
  «второстепенного контента» в мобильном UX).
- Подробный AI-разбор не генерируется заранее вслепую для каждого слова «на
  всякий случай» — только короткая версия сразу, подробная — по явному запросу
  (экономия и на стоимости AI-вызовов, и на визуальном шуме).

## 9. Открытые вопросы / нерешённые развилки (потенциально интересно для брейншторма)

- Как автоматически определять, какие слова/фразы в НОВОМ, ещё не написанном
  человеком тексте достойны аннотации (сейчас для входного пайплайна это решается
  генерацией аннотаций для всех значимых токенов без исключений — нет отдельного
  шага «отбора» важных слов).
- Единый провайдер текста/грамматики vs единый провайдер озвучки — сейчас это
  разные развилки выбора, не обязана быть одна и та же компания.
- Пороги quality gate (95% прямого покрытия / 10% доли восстановленных) —
  подобраны на глаз, не откалиброваны статистически; данные для калибровки
  копятся в `Lesson.alignmentReport` у каждого сгенерированного урока.
- Нет пока полноценной работы с личной библиотекой: сохранение — стаб на
  localStorage, нет прогресса изучения слов между уроками, нет статистики,
  нет spaced repetition поверх уже встреченных слов.
- Не решено, как выглядит вход «своя статья по ссылке» (URL) — сейчас только
  вставка текста руками или генерация по теме.
